What is TLS?

TLS (Transport Layer Security) is the cryptographic protocol that secures communications over a network. It replaced SSL and is the backbone of HTTPS, gRPC, Kafka, and virtually every modern secure channel.

```
🔐

Confidentiality

All data in transit is encrypted. Even if intercepted, it's unreadable without the session keys. Uses symmetric encryption (AES-GCM) after handshake.

✅

Authentication

Verifies you're talking to the real server (not an impersonator) via digital certificates signed by a trusted Certificate Authority (CA).

🛡️

Integrity

Ensures data isn't tampered in transit. Every record has a MAC (Message Authentication Code) — any modification is detected and the connection is dropped.

PKI — The Trust Chain

Public Key Infrastructure is the hierarchy that makes certificate trust work at internet scale.

Browser / Client

Root CA

  • Self-signed, kept offline (air-gapped)
  • Pre-installed in OS/browser trust stores
  • Signs Intermediate CAs only
  • Examples: DigiCert Global Root, ISRG Root X1 (Let's Encrypt)

Intermediate CA

  • Signed by Root CA, lives online
  • Issues end-entity (leaf) certificates
  • If compromised → only that intermediate is revoked
  • Must be included in server's cert chain

Leaf / Server Cert

  • Tied to a specific domain/IP/service
  • Contains server's public key
  • Validity: typically 90 days – 1 year (2025+: 47 days)
  • Signed by Intermediate CA

Internal / Private CA

  • Used in enterprises for internal services
  • Root distributed via MDM / GPO / Kubernetes ConfigMap
  • Examples: HashiCorp Vault PKI, AWS Private CA, cert-manager
  • Common in Citi-style BFSI environments
```
TLS Handshake

The negotiation that happens before a single byte of application data is exchanged. TLS 1.3 cut this from 2 round-trips to 1.

```

TLS 1.2 Handshake — 2 RTT

Client

→ ClientHello
supported TLS versions, cipher suites, random nonce, SNI
← Receive ServerHello
← Verify Certificate chain
→ ClientKeyExchange (pre-master secret)
→ ChangeCipherSpec + Finished
← Finished → 🔒 App Data
⟺
2 RTT

Server

← Receive ClientHello
→ ServerHello (chosen cipher)
→ Certificate + ServerKeyExchange
→ ServerHelloDone
← Receive ClientKeyExchange
→ ChangeCipherSpec + Finished

TLS 1.3 Handshake — 1 RTT

Client

→ ClientHello
includes key_share (DH params) upfront
← Derive shared secret immediately
← Verify encrypted certificate
→ Finished → 🔒 App Data
⟺
1 RTT

Server

← Receive ClientHello
→ ServerHello + key_share
→ Encrypted: Cert + CertVerify
→ Finished → 🔒 App Data
⚡
0-RTT Resumption TLS 1.3 supports 0-RTT session resumption using a pre-shared key from a previous session. Risk: replay attacks on non-idempotent requests.

Revocation Mechanisms

CRL — Cert Revocation List

A signed list published by CA of revoked serial numbers. Client downloads periodically. Can be large and stale.


⚠ Problem: Clients cache CRLs. A revoked cert may still work until CRL expires.

OCSP — Online Cert Status Protocol

Real-time query to CA's OCSP responder: "Is serial X:Y:Z still valid?" Returns: good / revoked / unknown


OCSP Stapling: Server queries OCSP, staples response to TLS handshake. Reduces latency and privacy leakage.

OpenSSL Commands

Essential commands for every DevOps / SRE engineer. These will get you through 95% of cert-related tasks.

```
// Generate private key + self-signed cert
``` # RSA 4096-bit key + self-signed cert (valid 365 days) openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes -subj ”/CN=myservice.example.com/O=MyCorp/C=IN” # ECDSA P-256 (faster, smaller, preferred for TLS 1.3) openssl ecparam -name prime256v1 -genkey -noout -out ec-key.pem openssl req -new -x509 -key ec-key.pem -out ec-cert.pem -days 365
```
// Generate CSR (to send to CA)
``` # Step 1: Generate private key openssl genrsa -out server.key 4096 # Step 2: Generate CSR (Certificate Signing Request) openssl req -new -key server.key -out server.csr -subj ”/CN=api.mybank.com/O=MyBank/OU=Infra/C=IN/ST=MH/L=Pune” # CSR with Subject Alternative Names (SAN) — REQUIRED for modern certs openssl req -new -key server.key -out server.csr -config san.cnf # see san.cnf below
```
// san.cnf — config for SAN
``` [req] distinguished_name = req_distinguished_name req_extensions = v3_req prompt = no [req_distinguished_name] CN = api.mybank.com O = MyBank C = IN [v3_req] subjectAltName = @alt_names [alt_names] DNS.1 = api.mybank.com DNS.2 = *.mybank.com IP.1 = 10.100.0.50
```
// Inspect / Verify certificates
``` # View cert details (human-readable) openssl x509 -in cert.pem -text -noout # Check expiry date only openssl x509 -in cert.pem -noout -dates notBefore=Jan 15 00:00:00 2024 GMT notAfter=Feb 15 23:59:59 2025 GMT # Check a remote server’s certificate openssl s_client -connect api.example.com:443 -servername api.example.com echo | openssl s_client -connect host:443 2>/dev/null | openssl x509 -noout -enddate # Verify cert chain openssl verify -CAfile ca-chain.pem server.crt server.crt: OK # Check if private key matches certificate openssl x509 -noout -modulus -in cert.pem | openssl md5 openssl rsa -noout -modulus -in key.pem | openssl md5 # MD5 values must match!
```
// Format conversions
``` # PEM → PKCS12 (pfx) — for Java/Windows openssl pkcs12 -export -out cert.pfx -inkey key.pem -in cert.pem -certfile ca-chain.pem # PKCS12 → PEM openssl pkcs12 -in cert.pfx -out cert.pem -nodes # DER → PEM openssl x509 -inform DER -in cert.der -out cert.pem # PEM → DER openssl x509 -outform DER -in cert.pem -out cert.der # Import into Java Keystore (JKS) — WebLogic / Kafka keytool -importkeystore -destkeystore keystore.jks -deststoretype JKS -srckeystore cert.pfx -srcstoretype PKCS12
```
// Batch expiry check script (production use)
``` #!/bin/bash — check expiry for all PEM files in a dir WARN_DAYS=30 for cert in /etc/ssl/certs/*.pem; do expiry=$(openssl x509 -enddate -noout -in “$cert” | cut -d= -f2) expiry_epoch=$(date -d “$expiry” +%s) now_epoch=$(date +%s) days_left=$(( (expiry_epoch - now_epoch) / 86400 )) if [ “$days_left” -lt “$WARN_DAYS” ]; then echo “⚠ $cert expires in $days_left days ($expiry)” fi done
Certificate Types

Certs are classified by validation level and coverage scope. Choosing wrong can block compliance audits.

```
By Validation Level

DV — Domain Validation

Proves control of the domain. Automated (ACME/Let's Encrypt). No company identity verified. Fastest (minutes). Fine for internal services, APIs.


Auto-renewable

OV — Organization Validation

CA verifies the organization is real (legal entity check). Shows O= in cert. Manual process, 1–3 days. Required by many banks and enterprises.


BFSI standard

EV — Extended Validation

Strictest vetting. CA verifies identity, legal existence, physical address. Green bar / padlock in old browsers. Now rarely visible. Used by banking portals.


Max trust
By Coverage Scope

Single Domain

Covers exactly one FQDN: api.example.com. Does NOT cover www.example.com.

Wildcard

Covers one subdomain level: *.example.com → api, www, app. Does NOT cover *.sub.example.com.


⚠ Wildcard key compromise → ALL subdomains compromised

Multi-Domain (SAN)

SAN cert lists multiple FQDNs, IPs, even wildcard entries. One cert for 100+ domains. Standard in enterprises.


✅ Preferred for microservice platforms

Specialized Cert Types

Code Signing Cert

Signs executables, scripts, packages. EKU = 1.3.6.1.5.5.7.3.3. Prevents tamper warnings. Required for signed JAR files, Windows drivers, CI/CD artifacts.

Client Auth Cert

EKU = 1.3.6.1.5.5.7.3.2. Used in mTLS — client presents this to server for authentication. Issued per-service or per-user in zero-trust arch.

Kubernetes Certs

  • apiserver.crt — kube-apiserver TLS
  • apiserver-kubelet-client.crt — apiserver → kubelet mTLS
  • etcd/server.crt — etcd cluster TLS
  • front-proxy-client.crt — aggregation layer
  • All managed by kubeadm / cert-manager

Kafka TLS Certs

  • Broker cert in JKS keystore
  • Truststore holds CA cert
  • Producer/consumer present client certs for mTLS
  • Config: ssl.keystore.location, ssl.truststore.location
```

Debug & Troubleshoot

Systematic approach to diagnosing TLS errors. 80% of issues fall into 5 categories.

```

SSL_ERROR_RX_RECORD_TOO_LONG

Client sent HTTPS to an HTTP port, or vice versa. Or wrong port. Check: is nginx listening on 443 with SSL config?

CERTIFICATE_VERIFY_FAILED

CA not trusted by client. Fix: add CA cert to client's truststore / OS trust store. Common with internal PKI.

SSL: CERTIFICATE_EXPIRED

Not After date is in the past. Renew immediately. Check ALL certs in the chain — intermediate can also expire.

Hostname Mismatch

CN or SAN doesn't match the hostname. Verify with: openssl x509 -in cert.pem -noout -text | grep DNS. Add missing SANs to CSR.

Incomplete Chain

Server sends leaf cert but not intermediate. Browser might work (downloads AIA), but curl/Java will fail. Always include full chain in ssl_certificate.

Wrong Cipher / Protocol

Server configured SSLv3/TLS1.0, client rejects it. Or ECDHE not supported. Use openssl s_client with -tls1_2 flag to test.

``` # Full handshake debug with TLS version and cipher used openssl s_client -connect host:443 -tls1_3 -servername host Protocol : TLSv1.3 Cipher : TLS_AES_256_GCM_SHA384 Verify return code: 0 (ok) # Test with curl (verbose TLS info) curl -v –tlsv1.2 https://api.example.com 2>&1 | grep -E “SSL|TLS|cert|expire” # Check which certs are being served (chain) openssl s_client -connect host:443 -showcerts 2>/dev/null | openssl x509 -noout -text # Verify OCSP response openssl ocsp -issuer chain.pem -cert server.crt -url http://ocsp.digicert.com -resp_text Response verify OK server.crt: good This Update: Jan 15 10:00:00 2025 GMT # Prometheus: scrape TLS cert expiry # x509-certificate-exporter or blackbox_exporter curl http://localhost:9115/probe?target=https://api.mybank.com&module=http_2xx
